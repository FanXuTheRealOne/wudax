# WUDAX 行中导航：暂停记录、底部工具栏与地图跟随性能 — 设计文档

日期：2026-07-11  
分支：main

## 1. 情境与目标

用户已到达路线起点，正在山野环境中单手查看 iPhone。阳光、汗水、弱 GPS 与有限注意力意味着行中页必须在一瞥内回答：**我是否仍在导航中、运动记录了多久、走了多远、该暂停还是继续？**

本轮解决两件相互关联的问题：

1. 地图在 GPS/罗盘高频更新下不应抢夺用户的缩放和拖动，也不应通过频繁重设视口造成卡顿或跳动。
2. 底部面板从“指标 + 详情/结束”重排为清晰的导航记录工具：导航状态、运动耗时、平均配速、里程与暂停/继续。

唯一主动作：记录中为“暂停”；暂停中为“继续行程”。结束不与暂停争夺同级操作。

## 2. 已确认的暂停语义

暂停不是关闭导航：

- 继续更新 GPS 位置、朝向、当前位置标记、地图与“是否在轨迹上”的视觉反馈；用户仍能缩放、拖动和重新定位地图。
- 冻结运动耗时、本次记录里程、平均配速、路线日志写入，以及基于运动时长推进的补水/疲劳/风险计时。
- 恢复后以累计的有效运动时长继续，不把暂停间隔计入配速、补水消耗或风险规则。
- 暂停期间的位置只服务于视觉导航，不追加到本次 `TrackRecorder`，也不将移动带入行程统计。

这是有意的手动记录暂停；自动暂停不属于本轮范围。

## 3. 信息架构与低保真布局

### 3.1 记录中

```
导航中 · 在轨迹上                              GPS 良好

00:48:12              12′40″/km               3.8 km
运动耗时               平均配速                 本次里程

剩余 8.6 km · 剩余爬升 540 m · 距日落 3.2 h

                 [  暂停  ]    [ 详情 ]
```

- 顶行是可读的状态句，不仅依赖绿色圆点；弱 GPS 时变为“定位不稳 · 精度约 XX m”。
- 三列数字的顺序固定：有效耗时、平均配速、本次里程。数字采用既有等宽数字字体，标签保留文字与单位。
- 次级行仅显示剩余距离、剩余爬升与日落安全窗口；ETA、定位细节、事件等进入详情，避免同屏仪表盘化。
- “暂停”是满宽主按钮的主要部分，使用高对比松针绿实色；“详情”是次级描边按钮。结束操作移入详情页，仍使用确认对话框。

### 3.2 已暂停

```
已暂停 · GPS 与地图仍在更新

00:48:12              12′40″/km               3.8 km
记录已冻结；恢复后继续累计

                 [  继续行程  ] [ 详情 ]
```

- 使用日照橙状态色 + 明确文案，不用持续闪烁或呼吸动画制造紧张。
- 所有运动数字保持静止；地图中的定位点仍可平滑移动。

### 3.3 风险与异常优先级

- 偏航：底部次级行替换为琥珀色实色提示“可能偏离路线 · 距路线约 X m”，保留暂停操作。
- 风险 check-in / 下撤：现有 `CheckinCardView`、下撤决策层位于最上层，不能被工具栏遮住或模糊成无法阅读的背景。
- 弱 GPS：显示精度与“数据仅作参考”；不伪装成准确轨迹。
- 无网：GPX、路线折线与既有记录照常工作；底图是否可见明确由系统缓存决定，不把在线瓦片表示为离线资源。
- 无定位、前往起点、完成行程：保持现有等待 GPS、虚线引导、行后复盘路径；仅记录中引入暂停/继续。

## 4. 地图性能与跟随策略

`TripDashboardView` 目前会响应 `LocationService.objectWillChange`，而 `RouteMapView` 的自动模式会随位置变化重置地图中心。即使单次操作不带动画，高频中心校正仍会和用户手势竞争，造成缩放“发涩”和位置跳动。

实现应遵守：

1. **手势优先**：用户捏合、拖动或旋转时立即停止自动跟随；只有点击定位按钮才恢复跟随。
2. **合并传感器更新**：将 SwiftUI 行中位置刷新限制在短、可感知的节流窗口内；朝向变化单独更新定位箭头，不触发相机或整条路线重建。
3. **安全区域跟随**：默认视口不因每次位置变化重居中；仅当用户移动超过合理阈值且接近可视区域边缘时，才平滑校正中心。首次定位/用户点击定位可以立即聚焦。
4. **缓存地图对象**：路线折线、端点、精度圈、虚线引导与标注都按输入变化增量更新；不得因秒表、Agent banner 或无关视图状态重建 `MKMapView`、路线 overlay 或相机。
5. **无动画叠加层，有限相机动画**：定位箭头可做短暂方向过渡；相机不对每一个 GPS 点做动画。路线聚焦和用户手动请求才允许明确、短时的相机转换。

性能验收以真实滑动和捏合为准：连续 10 秒位置/朝向更新期间，地图手势不被拉回、不掉帧；点击定位后只进行一次有意的回归定位。

## 5. 数据、SwiftUI 与测试对应

### `TripSession`

- 新增私有暂停记录：`isWorkoutPaused`、`pausedAt`、`accumulatedPausedDuration`，并提供 `pauseWorkout()` / `resumeWorkout()`。
- 有效耗时由开始时间扣除累计暂停时长得出；不要只修改显示层的 `TimelineView`。
- `evaluateActiveStatus()`、记录器追加、行程里程更新与风险时间规则在暂停时跳过；视觉定位/地图匹配可持续更新但不能写入统计。
- 结束行程前若处于暂停，应先固化暂停时长，保证持久化日志一致。

### `TripDashboardView`

- 将 `recordingStats` 改为“状态句 + 三项主指标 + 一行次级指标 + 暂停/详情动作”。
- `TimelineView` 只驱动有效耗时；平均配速格式化为 `mm′ss″/km`，在有效距离不足以可信计算时显示 `—`，而不是虚构速度。
- 暂停态显示“继续行程”，不让结束按钮出现在主操作位。

### `RouteMapView`

- 在 `Coordinator` 内记录最近一次有效自动跟随位置、时间和用户手势挂起状态。
- 位置点与朝向点分离：朝向刷新只更新 `UserNavigationPuckView`；相机重定位须经移动距离、时间和安全边界判断。
- 保持现有路线、引导线、精度圈与图层选择器；不改变底图离线能力的事实。

### 测试

- `TripSession`：暂停后有效时长不增加；恢复后只累计活动间隔；暂停期间记录器和里程不增加；结束于暂停态仍保存正确有效时长。
- 运动指标：平均配速在零/极小距离时为占位，已知时长和距离可得到正确格式。
- 地图策略：提取纯粹的跟随判定条件进行单元测试（首次定位、微小抖动、阈值移动、手势挂起、手动重新定位）。
- 现有全量单测、模拟器构建和真机手势检查不回归。

## 6. 视觉与可访问性

- 容器沿用暖纸白/深松针绿的稳定实色层级；不在长数据面板使用过度 Glass，不增加照片或抽象渐变。
- 暂停为深松针绿动作，暂停状态为日照橙提示；风险红仅留给真正的结束/危险决策。
- 主要点击目标不小于 48 pt，主暂停按钮目标高度约 52–56 pt；“结束”须通过详情页与确认对话框防误触。
- 支持 Dynamic Type：三项数字可缩放但标签不丢失，窄屏下次级信息可换行而不裁切；色彩之外始终有文字和 SF Symbol 状态说明。
- Reduce Motion 下关闭无关脉冲与数值切换动效；增强对比度时使用不透明的底部实色面板。

## 7. 参考、素材与非目标

- [AllTrails](https://apps.apple.com/us/app/alltrails-hike-bike-run/id405075943)：借鉴在轨/偏航与离线场景的状态优先级；不复制其品牌地图或控件外观。
- [Footpath](https://apps.apple.com/us/app/footpath-route-planner/id634845718)：借鉴离线 GPS 轨迹和速度/海拔数据组织；不复制其路线编辑交互。
- [Apple Fitness](https://support.apple.com/en-au/guide/iphone/iph8475d8510/ios)：借鉴暂停、恢复和结束的操作层级；不使用活动圆环或其视觉识别。
- [Apple Maps](https://support.apple.com/guide/iphone/set-your-location-and-map-view-iph10d7bdf26/218)：借鉴用户显式选择“跟随位置/跟随朝向”的心智模型；不照搬 Apple Maps 的导航界面。

行中页的素材是用户的真实地图、GPX 轨迹和当前位置，不引入网络图片或需要授权的装饰素材。

非目标：Live Activity、自动暂停、后台长时间位置记录优化、离线瓦片下载、路线规划与 Agent 会话改版。
