# WUDAX 离线徒步疲劳 Agent 设计说明

## 产品目标

WUDAX 在 iPhone 上完成一次徒步的三阶段闭环：行前理解用户与路线、行中主动守护、行后复盘与训练建议。核心判断离线运行，不依赖网络；端侧 Qwen3 模型负责自然语言引导和解释，确定性 Swift 规则负责事实计算、风险等级和受控行动。

## 三阶段流程

### Stage 1：连续聊天式行前计划

用户从首页点击“开始计划”后，进入一个连续滚动的 AI 聊天窗口。系统依次：

1. 主动请求 HealthKit 中与徒步相关的全部可读数据；拒绝或缺失时显示“未提供或暂无数据”，并用问卷补齐关键项。
2. 采集近期睡眠、疲劳、疼痛、疾病不适、训练与徒步经验、补水和补给习惯。
3. 通过系统文件选择器导入本次 GPX，解析轨迹、海拔、时间、停顿和数据质量。
4. 计算身体准备度、路线负荷、挑战差距、补给预算、装备建议和离线资源状态。
5. 在聊天流中吐出原生卡片，形成行前报告；完成出发门禁后进入 Stage 2。

计划轨迹和历史活动必须分开保存。历史 GPX 若被复制为计划，只保留几何与海拔，不沿用时间、速度或完成结果。

### Stage 2：主动 AI 向导

行中采用事件驱动评估：Core Location 更新、HealthKit 新样本、用户主观确认、前台最多每 30 秒一次节流重算。系统不能承诺 iOS 在后台或被终止时任意 30 秒唤醒；后台依赖持续定位和 HealthKit observer 的系统回调。

每次事件产生统一快照，包含数据来源、采样时间、新鲜度和可信度。确定性规则输出风险、原因和受控行动：继续、休息、喝水、降速、缩短路线、折返、检查安全。只有风险升级或冷却期结束时才发系统通知，避免轰炸。后台通知使用固定安全模板；端侧模型仅在可运行时补充解释，且不得降低规则风险或创造新行动。

### Stage 3：结果报告

结束行程后展示计划与实际对比、耗时和爬升、风险事件、补给与身体反馈、关键失控点。用户完成简短复盘后更新个人基线，并生成下一周期训练建议。原始轨迹和报告都保存在本机。

## 数据与隐私

- HealthKit：年龄范围、性别（如可用）、身高、体重、BMI、体脂、瘦体重、训练、路线、步数、步行跑步距离、爬楼、活动能量、锻炼时间、心率、静息心率、步行平均心率、HRV、血氧、呼吸频率、VO2 Max、睡眠及步态能力指标。
- 不请求临床病历、生殖健康、用药等与徒步准备无关的数据。
- HealthKit 每个值必须保留样本时间；锁屏不可用或数据过旧时降级可信度。
- GPX 原文件保存在 Application Support，SwiftData 只保存索引、摘要和报告；禁止把用户提供的原始 GPX 提交到 Git。
- Apple Watch 与 BLE 仅预留协议。本版本的 iPhone HealthKit 心率不是高频实时心率，界面必须如实表达。

## 后端边界与已实现能力

以下能力不需要后端，已经在 App 内实现：GPX 文件选择和容错解析、路线/海拔质量分析、Core Location 实时定位、GPX 轨迹叠加、轨迹记录、HealthKit observer、确定性风险规则、本地 Qwen3 工具调用、系统本地通知、行程和报告本地持久化。

“完整离线地图”需要额外的地图瓦片/矢量包授权、区域裁剪、版本签名、下载校验和资源更新策略。当前产品明确使用“仅路线离线模式”：GPX 线、海拔、当前位置和进度不依赖网络；MapKit 底图如果没有系统缓存可能为空，不把它伪装成完整离线底图。若产品要交付完整底图，需要接入 MapLibre/合法瓦片供应商或自有离线包分发服务，这属于地图资源供应链，不是徒步 Agent 后端 API。

主动式 AI 也不依赖后端：位置样本、HealthKit 新样本、前台 30 秒节流和用户状态确认都会进入同一套 Swift 风险编排，规则决定风险和行动，端侧模型只补充解释。iOS 被终止后的执行仍受系统后台定位、HealthKit observer 和通知策略约束，不能承诺任意间隔唤醒。

## 规则工具与模型边界

确定性工具：`analyze_gpx`、`calculate_route_load`、`calculate_user_readiness`、`calculate_challenge_gap`、`calculate_supply_budget`、`build_equipment_checklist`、`match_route_progress`、`evaluate_fatigue_risk`、`select_controlled_action`、`summarize_trip`、`update_personal_baseline`、`build_training_advice`。

工具返回结构化、可审计结果。Qwen3 通过原生 tool call 请求工具，宿主执行后把结果回传模型。若模型未加载、输出损坏或超时，固定模板仍能完整展示事实、风险和行动。

## GPX 容错

解析器按元素 local-name 工作，兼容不严格命名空间和厂商扩展；支持 track、segment、point、waypoint。统计同时保留厂商值和本地派生值，检测坐标缺失、时间倒退、超长停顿、重复点和异常速度。测试使用脱敏小型 fixture，不提交用户原始轨迹。

## 离线地图与出发门禁

MapLibre 离线资源以 GPX 走廊区域管理，展示下载进度、大小、更新时间和完整性。没有可配置地图源时提供明确的“仅路线离线模式”，只画本地 GPX 与定位，不静默假装底图已离线。出发前必须检查 HealthKit/问卷、GPX、定位权限、通知权限、离线资源和补给确认；风险项可要求用户显式确认。

## UI 设计硬约束

必须复用当前 WUDAX 设计系统，不重新设计品牌视觉：

- 只用现有 `WDColor`：墨松绿、深苔绿、苔面、宣纸白、雾灰绿、琥珀、朱砂、竹青和墨色。
- 复用 `WDFont`、`InkCard`、`PillButton`、`GhostButton`、`ContourBackground`、现有山水素材与海拔图。
- 顶部透明，不添加横向白条、分隔线或额外外框。
- 卡片保持 18pt 圆角、原阴影和原留白；除 GhostButton 与风险印章外不新增描边。
- Stage 1 的 HealthKit、问卷、GPX、报告和装备内容全部作为聊天流内卡片，不跳成风格不一致的表单页。
- 首页、计划、门禁、行中、复盘之间保持现有转场和交互质感。

## 验收条件

1. 从首页“开始计划”可完整走通三阶段，不依赖示例路线。
2. HealthKit 会主动请求权限，缺失数据有清晰降级路径。
3. 系统文件选择器可导入真实 GPX，并展示质量与路线摘要。
4. 报告、风险和行动来自可测试规则，模型失败时仍可用。
5. 行中定位与事件驱动风险评估可工作，通知有升级/冷却策略。
6. 行后生成复盘、基线更新和训练建议。
7. 界面只复用现有配色和组件，顶部没有白条。
